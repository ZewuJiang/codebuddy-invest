# 已知堵点与应对策略 — App版（v3.0）

> **用途**：投研鸭小程序数据生产过程中的已知堵点与应对方案。
> **核心原则**：先保真，再发布。核心行情缺失时宁可阻断，也不能用估算值、Mock值和新闻转述顶上。
> **v3.0 变更**：Fear & Greed 数据已从产品中移除（太散户），相关堵点标记废弃。

---

## 数据采集堵点

| # | 堵点 | 降级路径 | 频率 |
|---|------|---------|------|
| 1 | Google Finance 403/超时 | → web_search → 东方财富/StockAnalysis | 高 |
| 2 | sparkline 历史数据无法获取 | → **回到采集批次重试；仍失败 → 阻断发布（禁止估算）** | 高 |
| 3 | chartData 30天数据缺失 | → 回到对应数据源重采，**禁止线性插值或估算扩展** | 中 |
| 4 | metrics PE(TTM) 数据缺失 | → 标注"—"，不阻断主流程（PE 为辅助指标，非核心行情） | 中 |
| 5 | 某板块标的全部数据失败 | → 使用 stock-universe.md 中的备选标的替代 | 低 |
| 6 | 港股/A股数据获取困难 | → 东方财富/同花顺 → 智通财经 → web_search | 中 |
| 7 | 大宗期货 Google 不支持 | → OilPrice.com → 金投网 → web_search | 中 |
| **8** | **AkShare CNH（离岸人民币）历史序列失败** | → **阻断发布，不允许退化到 CNY=X（在岸）硬编码估算；<br>降级路径仅允许 CNY=X 实时价作为当日最后备选，且无法构成7天历史序列时必须阻断** | 中 |
| **9** | **北向资金净买额数据为空** | → **根本原因（非接口故障）：自2024-08-19起，沪深交易所永久停止实时披露北向资金净买额。<br>AkShare `stock_hsgt_fund_flow_summary_em` 净买额字段永久为空，无法修复。<br>正式替代方案：改用「外资动向」代理指标，以港股（恒生+恒生科技）均涨跌幅判断外资偏好，<br>由 `calc_foreign_capital_proxy()` 函数自动计算状态（绿/黄/红）。** | 永久性 |
| **10** | **日经225 yfinance 量级校验失败（价格超出 18000-55000 区间）** | → **切换 AkShare `futures_foreign_hist("N225")` 备用通道；仍失败 → 阻断发布** | 低 |
| **11** | **KOSPI yfinance 量级校验失败（价格超出 1500-4500 区间）** | → **标注"数据待核实"，不阻断整体流程；注：正常 KOSPI 综合指数约 2300-2600，若出现 5000+ 则为数据污染** | 低 |

| **13** | **actions.type 使用了 bullish/bearish** | → **根本原因：bullish/bearish 是方向判断，不是操作指令，用户看到"看涨/看跌"标签不知道该怎么操作。<br>必须改用具体操作动词：hold（持有）/ add（加仓）/ reduce（减仓）/ buy（买入）/ sell（卖出）/ watch（关注）/ hedge（对冲）/ stoploss（止损）。<br>content 字段同步写明：标的名称 + 操作条件 + 参考价位（参考 MD 报告风格）** | 高 |

| **14** | **来源链接跨媒体伪造（source 与 url 不对应）** | → **根本原因：Bloomberg/FT/WSJ 等付费墙媒体无公开链接，强行配 Yahoo Finance / 其他媒体链接会让用户点开后发现内容不匹配，严重损伤信任。<br>正确做法：只填 source，不填 url；前端自动显示灰色不可点样式（chain-link-no-url）。<br>禁止：source="Bloomberg" + url="yahoo.com/..."** | 高 |

| **15** | **coreJudgments.logic 使用段落散文（大老板快读体验差）** | → **根本原因：logic 字段是整张简报里信息密度最高、逻辑最复杂的区域；散文段落把触发原因、传导路径、结论混在一起，用户需要主动"解析"才能提取要点，认知负荷高。<br>**强制格式**：短句 + 箭头符号（→）三段式串联：`触发原因 → 传导路径 → 核心结论`<br>正确示例：`停火信号遭否认，IRGC升级威胁美科技设施 → 每多一周冲突油价+$2~4（高盛模型）→ 霍尔木兹持续封锁，油价中枢系统性上移`<br>错误示例（禁止）：`停火信号发出即被否认，IRGC升级威胁美国科技设施，谈判进程从乐观转入拉锯阶段；高盛模型显示每增加一周冲突布伦特将额外上涨2-4美元`（段落散文）<br>**前端渲染说明**：logic 文字直接顶格，无 💡 灯泡图标前缀（已从 briefing.wxml 移除）；箭头自带方向感，禁止加 bullet point（`•`）** | 高 |

| **16** | **takeaway 缺少【】关键词标红标注** | → **根本原因：前端 `parseTakeaway()` 用正则 `/【([^】]+)】/g` 提取被中文方括号【】包裹的文字，渲染为红色高亮（`.takeaway-highlight { color: #e74c3c; font-weight: 700 }`）。若 takeaway 文本中没有任何【】，整段文字显示为纯黑色，关键行动指令和核心条件被淹没，大老板无法一眼抓到重点。<br>**强制规则**：每条 takeaway 必须标红 3~5 个关键词，选择最核心的行动指令（如「以防御为主」「能源对冲仓位」）和关键条件/数据（如「VIX回落至20以下」「NFP数据」）。<br>正确示例：`今日【以防御为主】，持有【能源对冲仓位】，科技股等待【VIX回落至20以下】再增配，明日【NFP数据】或加剧波动。`<br>错误示例①（禁止）：`今日以防御为主，持有能源对冲仓位...`（无【】，全黑）<br>错误示例②（禁止）：`【今日】【以防御为主】，【持有】【能源】【对冲】【仓位】...`（标红≥6个，失去重点）** | 高 |

| **17** | **chain[].url 和 coreJudgments[].references 被漏填（可选字段偷懒空置）** | → **根本原因：`url` 和 `references` 在 Schema 中标记为可选字段（🔸），但「可选」的唯一合法理由是「付费墙媒体（Bloomberg/FT/WSJ）无公开链接」，绝对禁止因 AI 偷懒或遗忘而留空。这两个字段是简报页最核心的交互功能——用户点击来源才能核实信息真实性，漏填等于直接摧毁用户信任。<br>**强制规则（终审必查）**：<br>① `coreEvent.chain[]` 每一条：若 `source` 填了 Reuters/AP/MarketWatch/BullionVault 等**非付费墙媒体**，`url` 必须有真实可访问的 https 链接；不能留 `""` 或 `null`。Bloomberg/FT/WSJ 无公开链接时可留空，但必须在 source 字段里体现。<br>② `coreJudgments[]` 每一条：必须有 `references` 数组，且每个 reference 格式为 `{name, summary, url}`，url 同样遵守上述规则。references 为可选字段，但每次产出至少填 1 条（哪怕来源是公开新闻页）。<br>**终审自查口诀**：「有 source 必有 url（付费墙除外），有判断必有 references」<br>违规后果：简报链接点击无反应，等同于功能缺失，严重损伤用户体验和数据可信度。** | **高（新增 2026-04-03）** |

| **18** | **`refresh_verified_snapshot.py` API 失败但脚本仍阻断全流程（旧版 v1.1 行为）** | → **根本原因（已修复）**：v1.1 脚本在 API 校正失败时直接 `exit 1`，导致上传步骤也被阻断——小程序当天完全无数据，比「用 AI 估算的 sparkline 上线」更糟糕。<br>**v1.2（方案 Alpha）已修复**：第1步（API校正）改为软依赖，失败时打印警告并继续执行上传；第0步（JSON语法校验）和第2步（上传）仍为硬依赖。<br>**v2.0（方案A）进一步升级**：脚本只写 sparkline/chartData，其他所有字段由 AI 直接填写，脚本失败对数据完整性影响降到最低。 | **永久性（设计变更）** |

| **19** | **`refresh_verified_snapshot.py` 脚本边界混乱（方案A之前的历史问题）** | → **根本原因（方案A已彻底解决）**：v1.x 脚本覆盖范围过大（price/change/trafficLights/riskScore/gics/metrics/sectors等），导致每次版本迭代都要在「脚本覆盖范围」和「AI数据保护范围」之间找平衡，不断出现新的覆盖事故（v1.1/v1.2/v1.3 均有修复记录）。<br>**方案A（v2.0）从根本上解决**：脚本只写两个字段——`sparkline(7天)` 和 `chartData(30天)`。原因：<br>① 这两个字段是 API 的**唯一不可替代价值**（历史序列只能从 yfinance 实时拉取，AI 无法准确生成）<br>② 其他所有字段，AI+搜索的准确性不低于（甚至高于）脚本（尤其是 price/change/GICS 等）<br>③ trafficLights/riskScore 的阈值规则已写死在 SKILL.md，AI 按公式填写同样可重现、无主观误差<br>**脚本攻击面**：从覆盖 15+ 字段降到只写 2 个数组字段，出错范围缩小到「sparkline 数组内容不对」，且失败时有 AI 估算兜底，不影响小程序上线。 | **永久性（架构升级）** |

## JSON 生成堵点

| # | 堵点 | 解决方案 |
|---|------|---------|
| 1 | 某必填字段数据缺失 | 若为核心行情字段 → 阻断发布；若为辅助字段（PE等）→ 标注"—" |
| 2 | sparkline 全为 0 或相同值 | 重新采集，**禁止基于估算微调** |
| 3 | analysis 文本含 markdown | 运行文本清洗：去除 `**`、`|`、`>`、`- ` 等 |
| 4 | change 类型错误（string 而非 number） | 强制转换 `parseFloat(change)` |
| 5 | 枚举值越界 | 映射到最近的合法枚举值 |
| **6** | **metrics 6项结构不符（旧格式：30日区间/数据源）** | → **已统一为方案C：最新价/单日涨跌/7日涨跌/30日涨跌/PE(TTM)/综合评级** |

## 上传堵点

| # | 堵点 | 解决方案 |
|---|------|---------|
| 1 | access_token 获取失败 | 检查 WX_APPID/WX_APPSECRET 配置 |
| 2 | 云数据库写入失败 | JSON 文件保留在本地，下次可手动重传 |
| 3 | JSON 文件含特殊字符导致 query 解析失败 | upload_to_cloud.py 已内置 `_sanitize_for_query` 清洗 |
| 4 | 部分集合上传成功/部分失败 | 成功的数据生效，失败的保持旧数据 |
| **5** | **上传后云端字段缺失或 dataTime 不一致** | → **v1.1 新增 `verify_upload()` 回读校验；校验失败时打印具体缺失字段/数组长度异常** |

| **20** | **riskNote 风险提示为散文段落（大老板阅读体验差）** | → **根本原因：原 `riskNote` 是一整段30-100字散文，前端直接 `{{riskNote}}` 渲染为一大段黑压压文字，和上方 `marketSummaryPoints` 的 bullet point 格式形成强烈反差——上面一条一条扫读，到风险提示突然变成散文段，阅读节奏断裂。<br>**v3.1 修复**：`riskNote` 升级为 `riskPoints[]` 数组（2-4条），前端用 `wx:for` 循环渲染，每条带红色圆点 `•` 前缀；图标从 ⚠️ 更换为 🛡️（盾牌），与 🎯⚡🌡️🧠 实心彩色 emoji 风格统一，不与重点事件 🚨 重复。<br>保留旧 `riskNote` 字段作为 fallback，前端优先读 `riskPoints`，缺失时按句号自动拆分兜底。<br>**产出规则**：每条 riskPoints 只说一个风险点（15-50字），最后一条可以是行动建议；禁止一整段散文塞进单个数组元素** | **高（新增 2026-04-05）** |

| **21** | **`sentimentLabel` 使用了非标准枚举值（如「偏悲观」）** | → **根本原因**：AI 在 JSON 生成阶段按直觉写了「偏悲观」，而非 Schema 定义的标准枚举「偏恐惧」。6个合法枚举值为：`极度恐惧/偏恐惧/中性/偏贪婪/贪婪/极度贪婪`，与 CNN Fear & Greed 指数标签对齐。<br>**对应关系**（强制）：score 0-20→极度恐惧 / 21-40→偏恐惧 / 41-60→中性 / 61-75→偏贪婪 / 76-90→贪婪 / 91-100→极度贪婪<br>**典型事故（2026-04-03）**：sentimentScore=42 应对应「中性」，AI 却写了「偏悲观」（非法值），导致前端情绪标签显示异常。<br>**终审自查**：先确认 sentimentScore 数值，再按对应关系查表填入合法 sentimentLabel | **高（新增 2026-04-05）** |

| **22** | **`marketStatus` 休市日无合法枚举值** | → **根本原因**：v2.2及之前 Schema 只定义了4个枚举（美股交易中/美股已收盘/盘前交易/盘后交易），美国公共假日（如耶稣受难日、感恩节、圣诞节）不在任何枚举覆盖范围内，AI 被迫写「美股今日休市（耶稣受难日）」长字符串，触发致命错误#5（枚举越界）。<br>**v2.3 修复**：新增第5个枚举值 `美股休市`。假日当天 AI 直接填写 `\"美股休市\"`，不在后面加括号说明具体假日名称（假日名称可在 coreEvent.title 或 dataTime 中体现）。<br>**前端兼容**：前端 `format.js getMarketStatus()` 不受影响（该函数基于实时时间计算，不读 JSON 中的 marketStatus 字段）。JSON 中的 marketStatus 仅为快照参考，前端有独立判断逻辑 | **中（新增 2026-04-05）** |

| **23** | **`WATCHLIST_YF_MAP` 未及时同步新板块标的，导致新增标的 sparkline/chartData 永久为 AI 估算值** | → **根本原因**：每次 watchlist 板块结构调整（新增/删除标的）后，`refresh_verified_snapshot.py` 的 `WATCHLIST_AK_MAP` 必须同步更新，否则脚本根本不知道要下载这些标的的历史数据，AI 估算值会永久保留无法覆盖。<br>**v3.0（2026-04-05）**：映射表已从 yfinance ticker 格式改为 AkShare 格式。<br>**A股格式**：新浪源格式为 `sz+代码`（深交所）/ `sh+代码`（上交所），如 `sz300750`、`sh600519`。<br>**港股格式**：新浪源格式为 `0` 补齐5位，如 `00700`（腾讯）、`09988`（阿里）。<br>**失败安全机制保底**：任何标的下载失败 → 脚本打印警告并跳过，保留 AI 估算值，不阻断整体流程。 | **高（新增 2026-04-05）** |

| **24** | **AkShare 新浪源/东方财富源限流（v3.0 新增）** | → **根本原因**：AkShare 底层对接新浪财经和东方财富的公开接口，无官方 API Key，短时间高频请求会触发 IP 限流。<br>**新浪源限流阈值**：约 30-50 次/分钟连续请求后可能触发，恢复较快（秒级）。<br>**东方财富源限流阈值**：约 5-10 次/分钟连续请求后触发 ConnectionError，恢复较慢（3-5分钟）。<br>**v3.0 防限流策略**：①逐个调用 + 0.3秒 sleep 间隔（~40 标的总耗时约 20 秒）；②新浪源为主力（限流阈值高），东方财富源仅作 fallback（限流后不连续重试）；③任何标的双源均失败 → 跳过（保留 AI 估算值），不阻断流程。<br>**长期风险**：新浪财经近年在逐步关闭开放接口，未来可能需要切换数据源。东方财富源作为 fallback 保底。<br>**如果限流频繁触发**：可将 `SLEEP_INTERVAL` 从 0.3 增加到 0.5 或 1.0 秒。 | **中（新增 2026-04-05）** |

| **25** | **riskPoints 包含操作建议（v5.0 新增——职责混淆）** | → **根本原因**：风险模块（🛡️ 风险情绪）的职责是「告诉大老板有什么风险」，不是「告诉大老板该怎么做」。如果风险点里也放操作建议（如"建议维持6成仓位+能源对冲"），会与聪明钱建议模块的 actionHints 信息重复，且削弱三段式架构的清晰度。<br>**v5.0 强制规则**：`riskPoints` 每条只描述风险本身（是什么风险+触发条件+影响方向），**禁止包含任何操作建议**。操作建议统一在 `actionHints` 中覆盖。<br>**自查**：逐条检查 riskPoints，含"建议"/"操作"/"仓位"/"对冲"等操作性词汇的条目 → 删除或改写为纯风险描述。<br>**典型事故（2026-04-06）**：riskPoints 第4条"建议维持6成仓位+能源对冲，等CPI落地后再调整"——纯操作建议，与决策建议模块内容完全重复。 | **高（新增 2026-04-06）** |

| **26** | **actionHints 每天凑操作建议（价投风格违背）** | → **根本原因**：大老板是巴菲特/段永平式的低频价值投资者，不需要每天的买卖建议。旧 `actions.today/week` 结构天然鼓励 AI 每天"凑"出2-4条操作建议，导致建议质量下降、变成噪音。<br>**v5.0 强制规则**：`actionHints` 大多数日子应为**空数组 `[]`**——"没有操作建议"本身就是最好的建议。只在极少数高置信机会时才填入（如段永平实地调研信号、巴菲特大额建仓、极端市场机会等）。<br>**禁止凑数**：禁止写"维持不变"、"持续关注"、"等待xxx"等无信息量的废话填充 actionHints。<br>**买卖建议门槛极高**：type=buy/sell 只在极端机会或极端风险时使用。 | **高（新增 2026-04-06）** |

| **27** | **持仓数据凭 AI 记忆编造——致命级（v5.0 事故）** | → **根本原因**：`topHoldings`（briefing）和 `smartMoneyHoldings`（radar）的权重数据由 AI 凭训练数据中的模糊记忆填入，未查证权威数据源。AI 对具体数字的记忆天然不可靠，且持仓权重每季度变动。<br>**典型事故（2026-04-06）**：伯克希尔 AAPL 写28%（实际22.6%，差5.4个百分点）、AXP 写10%（实际20.5%，差10.5个百分点且排序颠倒）、段永平 briefing 漏掉 BRK.B(20.6%) 和 NVDA(7.7%) 两大核心持仓、ARK 旗舰 TOP5 全部写错（CRWV 根本不在 ARKK TOP10，将"每日交易标的"误当"持仓权重"）。<br>**v5.0 铁律**：<br>① **必须查证**：每次生产 topHoldings/smartMoneyHoldings 数据时，必须通过 web_search 查询权威数据源（13Radar/Whalewisdom/StockAnalysis 等），禁止凭记忆直接输出<br>② **严格按权重降序排列**：禁止凭直觉排序<br>③ **金融数据"宁缺毋错"**：查不到确切数据时标注"待更新"或省略该条目，绝不编造数字<br>④ **港股标的不适用13F**：泡泡玛特(9992.HK)等港股上市公司不会出现在美国SEC 13F披露中，正确的信息渠道是港交所权益披露或投资者公开言论 | **致命（新增 2026-04-06）** |

| **28** | **briefing 与 radar 持仓数据不一致（交叉校验缺失）** | → **根本原因**：briefing.json 的 `topHoldings` 和 radar.json 的 `smartMoneyHoldings` 是同一份数据的精简版和详版，但两者在不同阶段独立生成，没有强制交叉校验。结果两者数据各写各的——即使恰好数字接近也可能都是错的（"虚假一致性"）。<br>**典型事故（2026-04-06）**：briefing 伯克希尔 AAPL 写28%、radar 写28.1%，看似一致实则都是错的（实际22.6%）；briefing 段永平只列3只，radar 列了10只，信息量断层。<br>**v5.0 铁律**：briefing.topHoldings 必须是 radar.smartMoneyHoldings 的精简投影（取 TOP5 + 一行式概要），两者的标的、排序、权重数字必须完全对应。生产时先写 radar 详版，再从 radar 数据中提取 briefing 简版，禁止两者独立生成。 | **高（新增 2026-04-06）** |

| **29** | **sentimentScore 机械对齐 CNN Fear & Greed（v3.7 新增）** | → **根本原因**：CNN F&G 是散户情绪的滞后加权指标，对机构和价投参考价值有限。F&G 打包了 Put/Call 比、VIX、垃圾债利差等 7 个子指标，但这些底层数据本身比合成值更有参考价值。对价投来说，F&G=19「Extreme Fear」可能恰恰意味着买入机会——情绪标签天然带散户视角。<br>**强制规则**：`sentimentScore` 必须基于 VIX 水平、Put/Call 比、信用利差、期限溢价、资金流向等多维度底层数据**独立综合打分**，禁止简单对齐 F&G 值。sentimentScore 与 F&G 允许 ±15 分合理偏差。<br>**典型事故（2026-04-06）**：F&G=19（Extreme Fear），AI 最初考虑将 sentimentScore 从 25 下调至 15-20，但经审视发现 sentimentScore=25（偏恐惧）是综合多维度的合理独立判断，不应被单一 F&G 指标牵着走。 | **中（新增 2026-04-06）** |

| **30** | **coreEvent.title 信息堆砌（v3.7 新增）** | → **根本原因**：AI 倾向于在标题中塞入所有关键信息点（3-4 个），导致标题过长（30-40字）且没有重点。好的标题应聚焦 1-2 个最核心矛盾，与 takeaway 形成「标题→结论」的层次感。<br>**v3.7 规范**：`coreEvent.title` 从 30-80 字收紧为 **20-50 字**，只抓最核心的 1-2 个矛盾。<br>**典型事故（2026-04-06）**：原标题"伊朗战争阴影下美股反弹+油价暴涨，本周迎CPI与FOMC纪要双重考验"（33字，4个信息点）→ 精简为"美股反弹遇上油价暴涨——本周CPI将裁决反弹真伪"（22字，2个核心矛盾）。 | **中（新增 2026-04-06）** |

| **31** | **行情数据凭 AI 记忆编造（markets.json / radar.json 价格数据失真）——致命级** | → **根本原因**：AI 凭训练数据中的"模糊记忆"直接输出行情价格，没有从行情数据源实时采集验证。与堵点#27（持仓数据凭记忆编造）本质相同，但影响面更大——行情数据用户每天查看、直接感知，一旦数字明显偏离（如离岸人民币写7.28实际6.88，差0.4元/5.8%），用户信任直接崩塌。<br>**典型事故（2026-04-06）**：离岸人民币 CNH 在 markets.json 写 `"price": "7.2850"`，实际从英为财情查得真实值 6.8852——AI 记忆停留在 2022-2023 年人民币贬值周期（7.1-7.35 区间），但 2026 年 4 月已升值到 6.88。连锁错误：sparkline 7天全错（围绕 7.28 伪造）、change 方向相反（+0.05%实际-0.02%）、radar.json trafficLights 状态判定错误（7.285>7.15 判定 yellow，实际 6.885<7.15 应为 green）、riskScore 连锁计算偏差。<br>**为什么其他资产没出错**：标普、油价等变化显眼（$70→$109），搜索时自然会验证；离岸人民币变化"隐蔽"（7.28 vs 6.88 都在"看似合理"范围），未触发警觉。<br>**杜绝规则（v2.7 强制）**：<br>① **所有 markets.json / radar.json 中的 price/value 字段**，每次生产时**必须从实时行情源查证**（英为财情/东方财富/Google Finance/新浪财经），禁止凭记忆直接输出<br>② **trafficLights 七盏灯**：value 查证后，status 必须按阈值表机械判定（不能凭直觉），riskScore 按公式 `30 + Σ(green=0, yellow=10, red=20)` 机械计算<br>③ **sparkline**：由 `refresh_verified_snapshot.py` 脚本从 AkShare 实时拉取（方案A），AI 不应手动填写 sparkline<br>④ **交叉校验**：markets.json 中的 commodities 离岸人民币 price 必须与 radar.json trafficLights 离岸人民币 value 数值一致<br>⑤ **终审自查**：生产完成后，对 markets.json 的 6 大板块逐一扫描「这个价格是我查的还是凭记忆写的？」——任何凭记忆写的数字立即 web_search 验证 | **致命（新增 2026-04-06）** |

| **32** | **riskAdvice 包含操作建议+机械引用 F&G（v5.4 新增——与简报页去操作化原则不一致）** | → **根本原因**：riskAdvice 规范原第③条要求「给出具体仓位建议」，直接与简报页 v5.0 已固化的「去操作化」原则矛盾——riskPoints 已禁止操作建议（堵点#25），但 riskAdvice 的规范没同步更新，导致下次产出必然重蹈覆辙。同时原第②条要求「结合 fearGreed.value 给出当前情绪方向」鼓励机械引用 F&G 单一指标值，与 sentimentScore 独立判断原则（堵点#29）矛盾。<br>**典型事故（2026-04-06）**：riskAdvice 写「CNN F&G仅19（极度恐惧），建议维持5成仓位+能源对冲，等CPI落地后再评估」——①机械引用 F&G=19 ②含操作建议"维持5成仓位+能源对冲"。<br>**v5.4 修复**：①riskAdvice 只描述「风险在哪+严重性如何+关键催化剂何时」，不给操作建议（操作统一在 actionHints）；②禁止机械引用 F&G 单一指标值，可综合底层指标（VIX/Put-Call/信用利差）判断情绪方向；③正确示例更新为"黄金$4,627亮红灯（避险情绪极端），布伦特$109逼近红灯区间——若周四CPI确认通胀二次抬头，当前反弹基础将被抽空。"<br>**自查**：riskAdvice 含"建议"/"仓位"/"对冲"/"F&G仅XX"等词汇 → 重写 | **高（新增 2026-04-06）** |

| **33** | **sparkline 最后一个点与 price 数值不一致（markets.json / watchlist.json）** | → **根本原因**：sparkline 由脚本从 AkShare 拉取历史数据补全，price 由 AI 从行情源直采。两者数据来源不同、时间戳可能有差异（如脚本拉的是前一日收盘，AI 写的是盘中实时价），导致 sparkline[-1] 与 price 出现偏差。小程序前端 mini-chart 组件用 sparkline 画折线图，价格文字显示 price——两者不匹配时折线图终点与文字价格矛盾，用户一眼看出数据有问题。<br>**典型事故（2026-04-06）**：AAPL price 显示 `$223.45`，sparkline 最后一个点 `255.92`——差 14.5%，图表终点远高于显示价格，直接暴露数据不可靠。<br>**v4.4 校验规则**：<br>① 终审阶段逐个标的比对 `sparkline[-1]` 与 `price` 数值，偏差超过 1% 则标记异常<br>② 修正优先级：price 以行情源实时数据为准（更新鲜），sparkline 由脚本下次执行自动覆盖；如需立即修正，手动将 sparkline[-1] 更新为与 price 一致的值<br>③ 根本解决：脚本执行时间尽量接近 AI 数据生产时间，减少时间差 | **中（新增 2026-04-06）** |

## 端到端零堵点保障

1. 数据源异常 → 自动切换备选源（仅限同类直接行情源，禁止退化到估算）
2. sparkline 缺失 → **回采后重试，禁止估算**
3. 板块标的不足 → 自动使用备选标的
4. 某个JSON完全无法生成 → 标记 `_fallback: true`，不上传该集合
5. 上传失败 → JSON 文件保留，不影响其他集合
6. 上传成功 → `verify_upload()` 自动回读校验，发现异常立即告警
7. **任何环节不停下等用户**
