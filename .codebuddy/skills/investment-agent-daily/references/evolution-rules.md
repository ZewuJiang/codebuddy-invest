# Skill 自主进化规则（v1.0）

> **用途**：定义投资Agent每日策略简报Skill的自主进化机制——准入门禁、分级标准、回滚机制、反思维度。
> **核心原则**：**质量只升不降。进化是手段，高质量产出是目的。宁可不进化也不退化。**
> **创建日期**：2026-03-18
> **触发时机**：每次日报第四阶段交付完成后，自动进入第五阶段反思引擎。

---

## 一、进化哲学

> "大部分时间不动，但每次动都有确定性。" —— 巴菲特式进化

1. **每天都做反思**（1-2分钟的内省检查），但大多数时候结论是「本次无优化提案」
2. **有发现时才产生提案**，不为产出而产出
3. **宁可不进化也不退化** — 质量铁栅栏永远优先于进化欲望
4. **进化是渐进式的** — 可能很久才有一次微小的改进，这完全正常
5. **每次进化必须可追溯、可回滚** — 精确diff记录，确保任何不良变更可撤销

## 二、质量铁栅栏（进化准入门禁 — 最高优先级）

> **任何进化提案必须通过以下全部检验，否则直接拒绝。此门禁凌驾于一切进化欲望之上。**

### 2.1 三项强制检验

| # | 检验维度 | 检验问题 | 不通过 = 拒绝 |
|---|---------|---------|--------------|
| 1 | **准确性** | 此变更是否可能导致数据不准确？（包括引入未经验证的数据源、放松校验标准等） | 任何可能降低准确性的变更 → **直接拒绝** |
| 2 | **完整性** | 此变更是否可能导致覆盖范围缩小？（包括减少扫描环节、降低批次数等） | 任何可能缩小覆盖范围的变更 → **直接拒绝** |
| 3 | **实时性** | 此变更是否可能导致信息延迟？（包括减少搜索频次、跳过验证步骤等） | 任何可能增加延迟的变更 → **直接拒绝** |

### 2.2 通过条件

```
三项检验全部通过（不降级） + 至少一项有明确提升 → 允许进入分级评估
三项检验全部通过 + 无明确提升但有流程优化 → 允许但标记为"流程优化"
任一项不通过 → 直接拒绝，不进入分级评估
```

### 2.3 铁栅栏自检清单（进化提案自查）

每个进化提案生成后，必须回答以下问题：

| # | 自检问题 | 要求 |
|---|---------|------|
| 1 | 如果这个变更在3/3那次事故前就存在，会让事故更严重还是更轻？ | 必须是"更轻"或"无影响" |
| 2 | 如果大老板看到这个变更后的报告，会觉得质量提升了还是下降了？ | 必须是"提升"或"无感知变化" |
| 3 | 这个变更是否增加了使用训练数据的诱惑？ | 必须是"否" |
| 4 | 如果回滚这个变更，是否能在1分钟内完成？ | 必须是"是" |

## 三、反思维度（6个方面）

> 每次日报完成后，按以下6个维度逐一反思。不是每个维度都会有发现，大多数时候大多数维度的结论是"无异常"。

### 3.1 六维反思框架

| # | 反思维度 | 核心问题 | 可能的进化方向 | 反思耗时 |
|---|---------|---------|--------------|---------|
| 1 | **数据源效率** | 今天哪些数据源顺利？哪些403/超时了？有没有发现更稳定的备选源？ | 更新降级路径表、新增数据源URL模板 | ≤20秒 |
| 2 | **新闻源发现** | 今天搜索中是否偶然发现了新的高质量信息源？ | 新增媒体到media-watchlist三级雷达清单 | ≤20秒 |
| 3 | **分析框架优化** | 今天的分析对大老板的投资决策帮助大吗？结论是否足够可执行？有没有新的分析维度值得固化？ | 优化§1结论框架、新增分析模块 | ≤30秒 |
| 4 | **覆盖盲区** | 今天有什么重要事件差点漏掉？有什么重要标的/行业没覆盖到？ | 更新知识库标的清单、新增扫描环节 | ≤20秒 |
| 5 | **流程效率** | 哪个环节耗时最长？有没有可以优化的步骤？ | 优化批次并行策略、精简冗余步骤 | ≤15秒 |
| 6 | **格式与呈现** | 大老板消息文本的信息密度够不够？表格是否清晰？有没有更好的呈现方式？ | 微调格式规范 | ≤15秒 |

### 3.2 反思输出模板

每次反思后，按以下格式记录（无论是否有提案）：

```markdown
### 第五阶段：自主反思引擎

**反思时间**: YYYY-MM-DD HH:MM
**本次报告版本**: 投资Agent-每日策略简报-YYYYMMDD

| 维度 | 结论 |
|------|------|
| 数据源效率 | ✅ 正常 / ⚠️ [具体发现] |
| 新闻源发现 | ✅ 无新发现 / 💡 [具体发现] |
| 分析框架 | ✅ 正常 / 💡 [具体发现] |
| 覆盖盲区 | ✅ 无盲区 / ⚠️ [具体发现] |
| 流程效率 | ✅ 正常 / 💡 [具体发现] |
| 格式呈现 | ✅ 正常 / 💡 [具体发现] |

**进化提案**: 无 / 有（见下方）
```

## 四、进化分级标准

### 4.1 三级分类

#### 🟢 低风险（自动执行，记录到evolution-log.md）

> **定义**：纯增量变更，不修改任何已有规则，不可能导致质量下降。

| # | 允许的变更类型 | 示例 |
|---|--------------|------|
| 1 | 新增数据源到降级路径表（作为新的备选源） | "StockAnalysis比MarketWatch更稳定，新增为GICS ETF第二备选" |
| 2 | 新增媒体到media-watchlist三级雷达清单 | "发现Semafor的AI覆盖不错，加入三级雷达" |
| 3 | known-pitfalls.md新增堵点记录 | "Google Finance VIX页面改版，新增降级路径" |
| 4 | evolution-log.md记录观察/发现（不涉及规则变更） | "观察到CNBC财报覆盖深度提升" |
| 5 | data-collection-sop新增URL模板（不删除已有模板） | "新增VIX:INDEXCBOE的Google Finance URL" |
| 6 | 新增搜索关键词到batch扫描SOP（不删除已有关键词） | "batch 0a新增'Nikkei Asia markets'关键词" |

**自动执行流程**：
```
发现优化点 → 通过质量铁栅栏三项检验 → 确认为低风险 
→ 自动写入对应文件 + evolution-log.md记录（含精确diff）
→ 在日报末尾「进化提案」模块通知用户（已执行）
```

#### 🟡 中风险（日报末尾报告，默认生效机制）

> **定义**：修改已有配置的优先级/顺序，或新增固定模块/标的，但不触及核心规则。

| # | 变更类型 | 示例 |
|---|---------|------|
| 1 | 调整数据源优先级顺序 | "建议VIX首选从Google Finance调整为CBOE直连" |
| 2 | 媒体从三级升为二级 | "Semafor科技覆盖稳定高质量，建议从三级升为二级" |
| 3 | 新增覆盖标的到一级/二级关注清单 | "CRWV(GPU云)频繁出现在基金持仓，建议加入二级关注" |
| 4 | 报告格式微调（新增/调整固定模块） | "§5红绿灯新增'信用利差'指标" |
| 5 | 扫描批次的合并或重排 | "batch 0和0a可以合并为一次综合扫描" |
| 6 | 数据验证门禁新增检查项 | "第1.5阶段新增'加密市场24h成交量'验证项" |

**默认生效机制**：
```
发现优化点 → 通过质量铁栅栏三项检验 → 确认为中风险
→ 在日报末尾「进化提案」模块详细说明（含变更内容+理由+预期影响）
→ 标记"如无异议，下次执行时自动固化"
→ 用户不反对 → 下次日报执行前自动固化到对应文件 + evolution-log.md记录
→ 用户明确反对 → 标记为"已拒绝" + evolution-log.md记录拒绝原因
```

**默认生效的等待期**：从报告输出到下次日报执行之间。如果用户在此期间未表示反对，视为同意。

#### 🔴 高风险（必须用户明确确认）

> **定义**：触及核心规则、致命错误清单、五段式架构等Skill基石的变更。

| # | 变更类型 | 示例 |
|---|---------|------|
| 1 | 修改/删除已有核心规则（RULE ZERO~FIVE中任一条） | — |
| 2 | 修改/删除致命错误清单中的任何条目 | — |
| 3 | 修改报告五段式架构（§1-§5的结构定义） | — |
| 4 | 修改大老板消息文本格式 | — |
| 5 | 修改/删除终审清单中的任何条目 | — |
| 6 | 任何可能降低数据准确性/完整性/实时性的变更 | — |
| 7 | 删除已有数据源/媒体/标的（而非新增） | — |
| 8 | 修改禁用词清单（放宽限制方向） | — |

**强制确认机制**：
```
发现优化点 → 通过质量铁栅栏三项检验 → 确认为高风险
→ 在日报末尾「进化提案」模块详细说明（含变更内容+理由+影响评估+回滚方案）
→ 明确标记"需要您确认后才会执行"
→ 用户明确同意 → 执行变更 + evolution-log.md记录
→ 用户未回应/反对 → 不执行 + evolution-log.md记录为"待确认"/"已拒绝"
```

### 4.2 分级判断流程图

```
进化提案生成
  │
  ├─ 通过质量铁栅栏三项检验？
  │   ├─ 否 → 直接拒绝，记录到evolution-log（状态：❌ 被铁栅栏拦截）
  │   └─ 是 ↓
  │
  ├─ 是否触及核心规则/致命错误清单/五段式架构/消息文本格式/终审清单？
  │   ├─ 是 → 🔴 高风险 → 日报末尾报告，等待用户明确确认
  │   └─ 否 ↓
  │
  ├─ 是否修改已有配置的优先级/顺序/级别？或新增固定模块/标的？
  │   ├─ 是 → 🟡 中风险 → 日报末尾报告，默认生效
  │   └─ 否 ↓
  │
  └─ 纯增量新增（不修改已有内容）？
      ├─ 是 → 🟢 低风险 → 自动执行 + 日报末尾通知
      └─ 否 → 回到上层重新判断（保守原则：无法判断时升级为更高风险等级）
```

### 4.3 保守原则

**当分级判断存在模糊地带时，永远向上升级一级**：
- 不确定是低风险还是中风险 → 按中风险处理
- 不确定是中风险还是高风险 → 按高风险处理
- 不确定是否通过铁栅栏 → 按不通过处理

## 五、回滚机制

### 5.1 回滚设计原则

> **任何自动执行的进化变更，必须能在1分钟内完全撤销，恢复到变更前的状态。**

### 5.2 精确Diff记录

每次进化变更（无论级别），在evolution-log.md中必须记录精确diff：

```markdown
**精确Diff**:
- **文件**: references/known-pitfalls.md
- **位置**: §数据采集堵点 表格末尾
- **变更前**: （表格原最后一行）
- **变更后**: 新增行 `| 8 | VIX Google Finance改版 | → CBOE直连 → web_search "VIX close today" | 低 |`
- **回滚方式**: 删除known-pitfalls.md中#8行
```

### 5.3 回滚触发条件

| 触发方 | 触发条件 | 回滚流程 |
|--------|---------|---------|
| **用户主动** | 用户说"回滚XX变更"/"撤销上次进化" | 根据evolution-log.md的精确diff执行反向操作 |
| **系统自检** | 下次日报执行中发现某个已固化的进化导致了问题 | 在反思引擎中标记"需要回滚"，按中风险在日报末尾报告 |

### 5.4 回滚执行规范

```
1. 读取evolution-log.md找到目标变更记录
2. 根据精确diff执行反向操作
3. 在evolution-log.md中追加回滚记录：
   - 原变更ID
   - 回滚原因
   - 回滚时间
   - 回滚后状态确认
4. 更新进化统计（已回滚+1）
```

## 六、进化日志管理规范

### 6.1 日志文件位置

`references/evolution-log.md`

### 6.2 条目编号规则

格式：`#YYYYMMDD-NN`（日期+当日序号）
- 例：`#20260318-01` = 2026年3月18日第1条
- 序号从01开始，当日多条则递增

### 6.3 条目生命周期

```
观察 → 提案 → 评估（铁栅栏+分级） → 执行/拒绝/待确认 → [回滚]
```

### 6.4 状态标记

| 状态 | 含义 |
|------|------|
| ✅ 已固化 | 变更已执行并写入对应Skill文件 |
| ⏳ 待确认 | 中/高风险提案已报告，等待用户确认 |
| 🔄 默认生效中 | 中风险提案已报告，等待期内（用户未反对则下次自动固化） |
| ❌ 已拒绝 | 被铁栅栏拦截或用户明确拒绝 |
| ⏪ 已回滚 | 曾固化但已撤销 |
| 📝 仅记录 | 观察/发现，不涉及规则变更 |

### 6.5 日志清理

- 超过90天的"仅记录"条目可以归档到日志底部的"历史归档"区
- "已固化""已回滚""已拒绝"条目永久保留（审计轨迹）
- "待确认"超过30天未确认的条目自动标记为"已过期"

## 七、与现有机制的边界

### 7.1 进化机制 vs known-pitfalls.md

| 维度 | known-pitfalls.md | evolution-log.md |
|------|-------------------|------------------|
| **定位** | 已知问题+降级路径（当前状态） | 进化观察+提案+变更记录（变更历史） |
| **写入时机** | 遇到新堵点时直接记录 | 每次反思引擎发现优化点时记录 |
| **关系** | 进化机制可以触发向known-pitfalls新增条目（低风险自动执行） | evolution-log记录变更的来龙去脉 |

### 7.2 进化机制 vs SKILL.md Changelog

| 维度 | SKILL.md Changelog | evolution-log.md |
|------|-------------------|------------------|
| **定位** | 版本大事记（高层摘要） | 每条变更的详细记录（含diff） |
| **粒度** | 每个版本号一行 | 每个变更一个完整块 |
| **关系** | 重大进化固化后，在下次版本升级时汇总到Changelog | evolution-log是Changelog的详细源头 |

### 7.3 进化机制不覆盖的范围

以下变更**不走进化引擎**，仍按原有方式处理：
- **用户直接指令的修改**（如"把XX规则改成YY"）→ 直接执行，不走进化审批
- **Bug修复**（如发现某个URL失效需要更新）→ 直接修复，进化日志中记录为"Bug修复"
- **紧急热修复**（如日报执行中发现Skill规范有错误）→ 直接修复，事后补记

## 八、进化频率预期

| 进化类型 | 预期频率 | 说明 |
|----------|---------|------|
| 🟢 低风险 | 1-2次/月 | 偶尔发现新数据源或新堵点 |
| 🟡 中风险 | 1-2次/季度 | 较少，需要积累足够观察才会提出 |
| 🔴 高风险 | 极少 | 只有在发现重大结构性问题时才会提出 |
| 📝 仅记录 | 每次日报 | 每次都有反思记录，但大多数是"无异常" |
| **"无提案"** | **大多数日报** | **正常状态——没有进化不是问题** |

## 九、致命错误（进化机制专用 — 零容忍）

| # | 错误类型 | 后果 |
|---|----------|------|
| 1 | 未通过铁栅栏就执行了变更 | 立即回滚 + 升级为高风险重新评估 |
| 2 | 高风险变更未经用户确认就执行 | 立即回滚 + 在下次日报中道歉说明 |
| 3 | 回滚记录不完整（无精确diff） | 补全diff后才允许继续 |
| 4 | 进化变更导致日报数据质量下降 | 立即回滚 + 该提案永久标记为"已拒绝" |
| 5 | 为了产出进化提案而强行找问题 | 记录"无提案"即可，不强求 |
